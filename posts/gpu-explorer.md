I use ***[Compiler Explorer](https://godbolt.org/)*** constantly. It has some CUDA support — you can compile a kernel and see the PTX — but GPU programming today spans far more than CUDA on NVIDIA. There is HIP for AMD, ***[Triton](https://triton-lang.org/)*** as a Python-based DSL, frameworks like ***[ThunderKittens](https://hazyresearch.stanford.edu/blog/2024-05-12-tk)*** pushing C++20 with architecture-specific intrinsics, and multiple compilation targets across vendors. For all of this, there is no complete solution. GPU Explorer is an attempt to build one.

GPU compilation is more layered than CPU compilation. A CUDA kernel does not compile to a single target. It first becomes ***[PTX](https://docs.nvidia.com/cuda/parallel-thread-execution/)***, NVIDIA's intermediate representation — a virtual ISA, portable across architectures. PTX is then lowered to ***[SASS](https://docs.nvidia.com/cuda/cuda-binary-utilities/)***, the actual machine instructions for a specific GPU. These are fundamentally different representations. PTX abstracts; SASS is what the hardware executes. On AMD, the split is ***LLVM IR*** to ***GCN/RDNA assembly***. Same idea, different compilers, different representations. Understanding what the compiler did requires seeing both layers, correlated back to your source.

![GPU compilation layers — source to intermediate representation to machine code](/diagrams/gpu-compilation-layers.png)

The core problem is ***line mapping***. Displaying compiler output is trivial. Showing which output instructions came from which source line is where the real work is. GPU compilers emit debug information (`.loc` directives, `!DILocation` metadata, address-keyed line tables) that encodes source locations, but it is not designed for interactive use. The compiler interleaves instructions from different source lines due to scheduling. Function preambles correspond to no source line at all. SASS line information is keyed by instruction address, not position, and must be correlated from a separate tool. Each platform and representation uses a different debug format. GPU Explorer parses all of these and produces a unified source-to-output mapping that works across panes in real time.

![Line mapping — source lines map to non-contiguous, interleaved output lines](/diagrams/line-mapping.png)

The result is a ***three-pane interface***: source on the left, intermediate representation in the middle, machine code on the right. Hover over a line in any pane and the corresponding lines in the other two light up. Write `__syncthreads()` and see the `bar.sync` it becomes. Switch from Ampere to Hopper and watch new instructions appear. Toggle between -O0 and -O3 and see the compiler eliminate loads, unroll loops, reorder memory access. The connection between what you write and what the GPU executes becomes immediate.

GPU Explorer supports ***NVIDIA*** (CUDA to PTX + SASS), ***AMD*** (HIP to LLVM IR + GCN assembly), ***[Triton](https://triton-lang.org/)*** (Python to PTX/LLVM IR via JIT compilation), and ***[ThunderKittens](https://hazyresearch.stanford.edu/blog/2024-05-12-tk)*** (C++20 matrix operations with architecture-specific intrinsics). The same example kernels — vector addition, tiled matrix multiplication, warp reductions, bank-conflict-free transpose — are available in each language. This makes it straightforward to see how the same algorithm compiles across paradigms and hardware.

The value is ***building intuition***. Seeing the cost of an abstraction. Seeing whether shared memory loads and barriers ended up where you expected. Seeing what a warp shuffle compiles to. Seeing where NVIDIA and AMD diverge on the same HIP kernel. These are questions that are difficult to answer by reading documentation and easy to answer by looking at the compiler output, if the tooling makes it fast enough to be part of your workflow rather than a separate investigation. GPU Explorer compiles on every keystroke, caches results, and returns output in under a second. The goal is to make inspecting GPU compilation as routine as inspecting CPU compilation already is.
